Developing

Source control

We follow GDS guidance on maintaining version controlled code.

We use Git and Gitlab for production code. We are migrating our legacy codebase from Subversion.

Read more on Git setup

Development environment

We are agnostic on tool choice, but they must be configured to work well as a team (such as using agreed code styles). The most popular IDEs in use are Eclipse and IntelliJ. If you choose to use Eclipse, there is a pre-built version of Eclipse for Windows.

Documentation

README

We document our code with a README at the root of each code repository.

README should include:

  1. Description of the capabilities/features
  2. How to build
  3. How to configure
  4. How to run

Comments

We follow 'Clean Code' rules for comments:

  1. Always try to explain yourself in code.
  2. Don't be redundant.
  3. Don't add obvious noise. This includes 'bugfix #' comments
  4. Don't use closing brace comments.
  5. Don't comment out code. Just remove.
  6. Use as explanation of intent.
  7. Use as clarification of code.
  8. Use as warning of consequences.

Bugs

When we get things wrong, defects are raised in Jira.

We try to fix bugs within the same sprint. This is to avoid an ever growing backlog of bugs: Fix it or cancel as low-priority/won't fix.

Technical debt

We sometimes have to get things out of the door when we're not completely happy with the way it's working. When this happens we record it as technical debt.

Continuous Integration

We perform Continuous Integration builds on every push to a feature branch.

A CI build will:

  • Compile
  • Unit Test (Maven Surefire)
  • Integration Test (Maven Failsafe)
  • Enforce automated code quality rules

We use Gitlab-CI for our build pipeline. This allows us to run downstream dependencies such as a database in a Docker container.

Automated code quality checks

We use SonarQube to enforce code quality according to static analysis and code test coverage quality gate.
